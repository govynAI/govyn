**GDPR Storage Region:**

1. **Update `LoggingConfig` in `src/types.ts`:** Add a `storageRegion` field:
   ```typescript
   /** Storage region for GDPR compliance: 'eu', 'us', or 'auto' (default: 'auto') */
   storageRegion: 'eu' | 'us' | 'auto';
   ```

2. **Update `src/config.ts`:** Parse `storage_region` from the YAML logging section. Default to `'auto'` if not specified. Validate that the value is one of `'eu'`, `'us'`, or `'auto'`.

3. **Update `LogEntry` in `src/types.ts`:** Add a `storage_region` field:
   ```typescript
   /** Storage region where this log entry is stored */
   storage_region: 'eu' | 'us' | 'auto';
   ```

4. **Update `ActionLogger` in `src/action-logger.ts`:** When creating a log entry, set `storage_region` from the logger's config. This metadata travels with the log entry so downstream systems (future cloud dashboard, R2 region hints) can route accordingly.

5. **Update default config in `src/index.ts`:** Add `storageRegion: 'auto'` to the default LoggingConfig.

**Log Purge Endpoint:**

6. **Add `purgeBefore(date: Date): { deletedLogs: number; deletedPayloads: number }` method to `ActionLogger` in `src/action-logger.ts`:**
   - Read the current JSONL log file and all rotated log files in the directory
   - For each file: read line by line, parse JSON, check timestamp against the `date` parameter
   - Rewrite each file with only entries that have timestamps >= the date (keep recent, delete old)
   - For removed entries that have `has_payload: true` and `payload_id`, delete the corresponding payload file from the payloads directory
   - Return the count of deleted log entries and deleted payload files
   - Handle gzipped rotated files: decompress, filter, if all entries removed then delete the file; if some remain, recompress

7. **Add DELETE handler in `src/log-api.ts`:** Handle `DELETE /api/logs?before=DATE`:
   - Parse `before` query parameter as ISO 8601 date string
   - Validate it's a valid date; return 400 if invalid
   - Call `actionLogger.purgeBefore(new Date(before))`
   - Return JSON: `{ "success": true, "deleted_logs": N, "deleted_payloads": M }`
   - If `before` parameter is missing, return 400: "Missing required 'before' query parameter"

8. **Update `src/server.ts`:** Ensure `handleLogApi` is called for DELETE method on `/api/logs`. Currently the router checks `url.startsWith('/api/logs')` which should match, but verify the handler dispatches DELETE properly.

**Tests:**

9. **Create `tests/unit/gdpr-config.test.ts`:**
   - Test config parsing with `storage_region: eu` -> verify `storageRegion: 'eu'` in LoggingConfig
   - Test config parsing with `storage_region: us` -> verify `storageRegion: 'us'`
   - Test config parsing with no storage_region -> verify default `'auto'`
   - Test config parsing with invalid storage_region value -> verify error or fallback
   - Test that ActionLogger includes `storage_region` in every log entry

10. **Create `tests/integration/log-purge.test.ts`:**

    - Call `DELETE /api/logs?before=invalid-date` -> verify 400 error
    - Test with full-payload mode: send request, wait for flush, purge, verify payload file also deleted

**Load Test:**

Create `tests/load/load-test.ts` — a Vitest test file (not a standalone script) so it runs in CI:

1. Start a mock upstream server that responds with a canned OpenAI JSON response (200 OK, ~500 bytes, 10ms artificial delay to simulate minimal upstream processing).
2. Start govyn proxy pointing to mock upstream with all features enabled: cost tracking, budget enforcement (set limits high so nothing blocks), loop detection (threshold=1000 so it doesn't trigger), and action logging to temp directory.
3. Define the load test:
   - Create 100 concurrent requests using `Promise.all()` — each makes an HTTP request to the proxy `/v1/openai/v1/chat/completions` with a unique agent ID suffix (to avoid loop detection) and a small JSON body.
   - For each request, record: start time, time to first byte, total time.
   - After all complete, calculate:
     - p50, p95, p99 latency (sort times, pick percentile indices)
     - Mean latency
     - Error count (any non-200 responses)
     - Throughput (requests/second)
4. Assertions:
   - All 100 requests return 200
   - p95 total latency overhead (proxy latency minus mock upstream's 10ms) is under 50ms
   - No request corruption: each response body is valid JSON matching expected structure
   - Zero errors
5. Use `test.concurrent` or a single test with 100 parallel requests. Set timeout to 30 seconds for the test.

**Failure Mode Tests:**

Create `tests/failure/fail-open.test.ts`:

1. **Log directory unavailable:** Start proxy with logging configured to a non-writable or nonexistent directory (e.g., `/nonexistent/path/logs` on Linux or a path that will fail). Send a request through the proxy. Verify:
   - The request is still forwarded to upstream successfully (proxy doesn't crash)
   - The response is correct (200 with expected body)
   - A warning is logged to stderr/console about logging failure
   - This tests ADR-002 fail-open behavior for logging subsystem

2. **Cost aggregator overflow:** Send 10,000 cost records to the aggregator, then query costs. Verify the proxy continues responding (doesn't run out of memory or hang for a reasonable record count).

3. **Budget enforcer with no config:** Start proxy with empty budgets map. Send requests. Verify all pass through (no false blocks).

Create `tests/failure/graceful-degradation.test.ts`:

1. **Corrupt config values:** Create a config YAML with invalid budget values (negative numbers, NaN strings). Load it with `loadConfig()`. Verify either:
   - It throws a clear error (preferred for obviously invalid config), OR
   - It falls back to safe defaults with a warning

2. **Upstream timeout:** Start mock upstream that never responds (hangs). Set a client-side timeout of 5 seconds on the proxy request. Verify the proxy:
   - Eventually returns an error to the client (502 or timeout error)
   - Does not leak the connection
   - Remains responsive for subsequent requests to a healthy upstream

3. **Rapid server start/stop:** Start and stop the proxy 5 times in succession. Verify no port conflicts, no leaked listeners, clean shutdown each time.
