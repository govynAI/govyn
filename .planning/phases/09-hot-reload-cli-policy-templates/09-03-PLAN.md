Create directory `templates/policies/` and write 11 YAML policy files. Each file MUST use `version: 1` and a `policies:` array. Include descriptive comments explaining the use case, how to customize, and what each policy does.

**1. production-safety.yaml** (type: block, regex mode)
- Block destructive SQL patterns: `DELETE\s+FROM`, `DROP\s+TABLE`, `TRUNCATE`, `ALTER\s+TABLE.*DROP`
- Block dangerous shell commands: `rm\s+-rf`, `mkfs`, `dd\s+if=`
- Block credential exposure: patterns matching API keys, tokens in request body
- Scope: global, enabled: true
- Use `regex: true` in match criteria, match on body

**2. budget-control.yaml** (type: budget_limit)
- Two policies: daily limit ($50/agent) and monthly limit ($500/agent)
- Scope: global (applies to all agents — users customize by changing scope to per-agent)
- Include comments showing how to customize: per-agent scope example

**3. pii-protection.yaml** (type: content_filter)
- Patterns: ssn, credit_card, email, phone (all built-in pattern names)
- reveal_pattern: false (secure default — don't leak what was detected)
- Scope: global
- Include comment explaining built-in vs custom patterns

**4. business-hours-only.yaml** (type: time_window)
- Allow mode: weekdays 09:00-17:00
- Timezone: UTC (with comment showing how to change to America/New_York etc.)
- Days: weekdays preset
- Scope: global

**5. read-only-mode.yaml** (type: block)
- Block POST, PUT, DELETE, PATCH methods via body/path heuristics
- Actually: block requests whose path contains write-oriented endpoints — use regex on path matching patterns like `/delete`, `/create`, `/update`, `/write`, `/modify`
- Alternative approach: block by action_type for known APIs. Include two policies: one blocking by action type, one by path pattern
- Scope: global

**6. emergency-lockdown.yaml** (type: block)
- Single unconditional block policy (no match criteria = blocks everything)
- Scope: global
- enabled: false by default (operator enables during emergency)
- Include comment: "Enable this policy to immediately block ALL agent API requests"

**7. smart-model-routing.yaml** (type: model_route) — THE SHOWCASE TEMPLATE per ADR-021
- Two model_route policies: one for Anthropic, one for OpenAI
- **Anthropic policy:**
  - model_aliases: { cheap: "claude-haiku-4-20250514", standard: "claude-sonnet-4-20250514", premium: "claude-opus-4-0520" }
  - Rules (in priority order):
    1. Short prompts (<500 tokens) -> cheap (Haiku)
    2. No complex keywords (no "analyze", "reason", "explain in detail", "compare") -> cheap
    3. System prompt contains "summarize", "translate", "format", "extract" -> cheap
    4. Tool calls present -> standard (Sonnet, needs function calling reliability)
    5. Long prompts (>4000 tokens) -> standard (complex but not premium-worthy)
    6. Default passthrough (keeps original model)
  - max_downgrade_level: standard (never route below standard for safety)
  - Scope: target: anthropic
- **OpenAI policy:**
  - model_aliases: { cheap: "gpt-4o-mini", standard: "gpt-4o", premium: "gpt-4o" }
  - Rules: similar pattern — short/simple -> gpt-4o-mini, complex -> gpt-4o, default passthrough
  - Scope: target: openai
- Include comments with savings estimate: "Typically saves 40-70% on LLM costs by routing simple tasks to cheaper models"

**8. rate-limit-standard.yaml** (type: rate_limit)
- 10 requests per 60 seconds per agent
- Scope: global
- Include comment showing how to customize per-agent

**9. cost-conscious.yaml** (combined: model_route + budget_limit)
- Model routing to prefer cheap models (aggressive routing)
- Budget limit: $20/day, $200/month
- This template combines two policy types to maximize cost savings

**10. development-sandbox.yaml** (type: rate_limit, very permissive)
- High rate limit: 100 requests per 60 seconds (permissive for dev)
- Budget limit: $100/day (generous but not unlimited)
- Include comment: "For development/testing — swap to production-safety.yaml before deploying"

**11. high-security.yaml** (combined: content_filter + time_window + block)
- PII content filter (all built-in patterns + custom regex for API keys)
- Business hours time window (weekdays 08:00-18:00)
- Block known dangerous patterns (production-safety style)
- This is the "everything on" template for maximum governance

**All templates must:**
- Start with `version: 1`
- Have a `policies:` array
- Include YAML comments explaining purpose and customization
- Use valid policy schema that parsePoliciesFromFile accepts

  <action>
Create `tests/policy-templates.test.ts` with two test suites:

**Suite 1: "Template validation" — all templates parse without errors**

Use `fs.readdirSync('templates/policies')` to get all YAML files. For each file:
- Call `parsePoliciesFromFile(path)`
- Assert `result.success === true`
- Assert `result.policies.length > 0`
- Assert `result.errors.length === 0`

This should be a parameterized test using `describe.each` or a loop so adding new templates auto-tests them.

**Suite 2: "Template evaluation" — each template evaluates correctly against sample requests**

Test each template by loading into a PolicyEngine and evaluating with appropriate mock request contexts:

1. **production-safety:** Request with body containing "DELETE FROM users" -> denied. Normal request -> allowed.
2. **budget-control:** Use CostAggregator with $60 recorded spend -> denied (daily $50 limit exceeded). With $30 spend -> allowed.
3. **pii-protection:** Request body with "123-45-6789" (SSN pattern) -> denied. Normal text -> allowed.
4. **business-hours-only:** Evaluate at Tuesday 10:00 UTC -> allowed. Evaluate at Tuesday 22:00 UTC -> denied. Use injectable `now` timestamp via EvaluateOptions for deterministic testing.
5. **read-only-mode:** Request with path containing "/delete" -> denied. Request with path "/chat/completions" -> allowed (read-only endpoints pass).
6. **emergency-lockdown:** Template is disabled by default -> all requests allowed. Enable the policy programmatically -> all requests denied.
7. **smart-model-routing:** Anthropic request with <500 token estimate -> routeTo is haiku model. Request with >4000 tokens -> routeTo is sonnet. Normal request -> passthrough (model unchanged). OpenAI request with short prompt -> gpt-4o-mini.
8. **rate-limit-standard:** Send 10 requests -> all allowed. 11th request -> denied with retryAfterSeconds.
9. **cost-conscious:** Short prompt -> routes to cheap model. Over daily budget -> denied.
10. **development-sandbox:** 50 requests within 60 seconds -> all allowed (high rate limit).
11. **high-security:** Request with PII -> denied. Request outside hours -> denied. Request with dangerous pattern -> denied. Clean request during hours -> allowed.

**Setup pattern for each test:**
```typescript
const engine = new PolicyEngine();
const result = engine.loadFromFile('templates/policies/{template}.yaml');
expect(result.success).toBe(true);
// For budget tests: engine.setCostAggregator(aggregator);
const evalResult = engine.evaluate(context, options);
```

**Important:** For time-based tests, use `EvaluateOptions.now` to inject deterministic timestamps. For budget tests, pre-populate the CostAggregator with mock cost records.
