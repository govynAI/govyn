<objective>
Add SSE streaming passthrough, YAML-driven configuration, a health check endpoint, and upstream error forwarding to the proxy server.

Purpose: Streaming is how 90%+ of production LLM calls work (ADR-005). Without SSE passthrough, the proxy is unusable for real workloads. YAML config replaces the hardcoded defaults from Plan 01-01. The health endpoint enables monitoring. 429 forwarding ensures agents see real rate limits from upstream providers (ADR-016).

Output: A fully functional proxy that streams SSE responses chunk-by-chunk, loads configuration from YAML, exposes a health endpoint, and transparently forwards upstream errors including 429s with rate-limit headers preserved.
</objective>

<task type="auto">
  <name>Task 2: SSE streaming passthrough and upstream 429 error forwarding</name>

  <action>
    Create src/streaming.ts — SSE streaming passthrough:
    - Export `handleStreamingResponse(upstreamRes: IncomingMessage, clientRes: ServerResponse)`
    - Detect SSE response by checking upstream Content-Type contains "text/event-stream"
    - Set response headers on clientRes: Content-Type: text/event-stream, Cache-Control: no-cache, Connection: keep-alive
    - Copy the upstream status code to client response
    - Use Node.js stream piping: `upstreamRes.pipe(clientRes)` — this is the fastest path for chunk-by-chunk forwarding without buffering
    - CRITICAL: Do NOT accumulate chunks in memory. Do NOT wait for the full response. The pipe call handles backpressure automatically.
    - Handle upstream connection close: when upstream ends, ensure client response also ends cleanly
    - Handle client disconnect: if client disconnects early, destroy the upstream connection to avoid resource leaks
    - Record timestamp when first data chunk arrives from upstream (for latency measurement per PRXY-06)

    Update src/proxy.ts — integrate streaming and error forwarding:
    - After sending the request to upstream and receiving the response:
      1. Check if response Content-Type is text/event-stream AND the request indicated streaming (or just trust the upstream response type)
      2. If SSE: call handleStreamingResponse to pipe the stream
      3. If not SSE: pipe normally as before (non-streaming response)
    - For ALL upstream responses (including errors):
      - Forward the exact status code from upstream to client
      - Forward ALL headers from upstream to client, especially rate-limit headers: Retry-After, x-ratelimit-limit-requests, x-ratelimit-limit-tokens, x-ratelimit-remaining-requests, x-ratelimit-remaining-tokens, x-ratelimit-reset-requests, x-ratelimit-reset-tokens
      - Do NOT intercept or modify 429 responses — pass them through verbatim (per ADR-016)
      - Do NOT retry on 429 — the agent or framework handles retry logic
    - For proxy's own errors (upstream unreachable, connection timeout), continue to return 502 with Govyn error format

    Create tests/streaming.test.ts:
    - Create a local test HTTP server that emits SSE events (data: chunks with delays between them)
    - Send a request through the proxy to this test server
    - Verify that chunks arrive at the client as they are emitted (not all at once at the end)
    - Measure time between test server emitting first chunk and client receiving first chunk — must be <50ms
    - Verify Content-Type: text/event-stream is set on client response
    - Test client disconnect: disconnect client mid-stream, verify no crash and upstream connection is cleaned up
