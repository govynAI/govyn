**3. Create `src/loop-detector.ts`:**

Create LoopDetector class:
- Constructor takes `defaultConfig: LoopDetectionConfig` (used when agent has no per-agent override), `agentConfigs: Map<string, AgentConfig>` (to read per-agent overrides)
- Internal state: `Map<string, { hash: string; timestamps: number[] }[]>` — per-agent, per-request-key sliding window of timestamps. The request key is a combination of the target endpoint path and a hash of the request body.
- `recordRequest(agentId: string, endpoint: string, bodyHash: string): void` — record a timestamp for this agent+endpoint+bodyHash combination. Prune timestamps outside the window.
- `isLooping(agentId: string, endpoint: string, bodyHash: string): boolean` — after recording, check if the count of timestamps within the window >= threshold for this agent. Use the agent's per-agent config if available, otherwise the default.
- `getRequestHash(body: Buffer): string` — compute a simple hash of the request body. Use Node.js `crypto.createHash('sha256').update(body).digest('hex').slice(0, 16)` for a compact hash.
- `clear(agentId?: string): void` — clear tracking data for one or all agents.

Sliding window implementation: for each agent+key, keep an array of timestamps. On `recordRequest`, push the current timestamp, then filter out timestamps older than windowSeconds. On `isLooping`, check if the filtered array length >= threshold.

**4. Update `src/budget-enforcer.ts`:**

Add block/unblock state management:
- Add private `blockedAgents: Map<string, { reason: string; expiresAt: number }>` — tracks agents that are temporarily blocked (by loop detection or other means).
- `blockAgent(agentId: string, reason: string, cooldownSeconds: number): void` — adds entry to blockedAgents with `expiresAt = Date.now() + cooldownSeconds * 1000`.
- `unblockAgent(agentId: string): boolean` — removes entry from blockedAgents. Returns true if the agent was actually blocked.
- `isBlocked(agentId: string): { blocked: boolean; reason?: string; expiresAt?: number }` — check if agent is in blockedAgents AND the expiry hasn't passed. If expired, auto-remove and return `{ blocked: false }`.
- Update `checkBudget()` to FIRST check `isBlocked()` — if blocked, return `{ allowed: false, code: 'loop_detected' }` (or whatever the original block reason was). This ensures loop-blocked agents stay blocked for the cooldown even if their budget is fine.

**1. Update `src/server.ts`:**

- Import LoopDetector
- Update `startServer()` signature to accept LoopDetector as a fourth parameter: `startServer(config, aggregator, budgetEnforcer, loopDetector)`
- Add route for `POST /api/agents/:agentId/unblock` — parse agentId from the URL path (regex or string split on `/api/agents/` and `/unblock`). Call `budgetEnforcer.unblockAgent(agentId)`. Return 200 with `{ success: true, agent_id: agentId }` if unblocked. Return 404 if agent was not blocked.
- In the proxy request flow, AFTER resolving the agent identity and BEFORE calling `budgetEnforcer.checkBudget()`:
  1. Read the request body (it's already buffered in proxy.ts, but we need the hash in server.ts for loop detection). Since the body is read in forwardRequest(), we need to restructure slightly: compute the body hash in the server request handler using the raw URL path and a hash of the buffered body.

  **Alternative approach (simpler):** Pass the LoopDetector into forwardRequest() and have it do loop checking after reading the body. This keeps the body-reading logic in one place. Add `loopDetector` and `budgetEnforcer` parameters to `forwardRequest()`. After reading the body buffer, compute the hash, call `loopDetector.recordRequest()` and `loopDetector.isLooping()`. If looping, call `budgetEnforcer.blockAgent()` with the agent's cooldown config, then return 429 with loop_detected error. This is cleaner than duplicating body reading in server.ts.

  **Chosen approach:** Move loop detection into `forwardRequest()` in proxy.ts. The budget check (for spending limits) stays in server.ts since it doesn't need the body. Loop detection goes in proxy.ts since it needs the body hash.

  So the flow becomes:
  1. server.ts: resolve agent → checkBudget() (blocks if over budget or in cooldown) → if allowed, call forwardRequest()
  2. proxy.ts: read body → compute hash → loopDetector.recordRequest() → loopDetector.isLooping() → if looping, blockAgent() and return 429 → otherwise forward normally

**4. Add budget reset scheduling to `src/budget-enforcer.ts`:**

Add a `startResetTimer(): void` method that:
- Calculates ms until next midnight UTC using `nextMidnightUTC()`
- Sets a `setTimeout` to run at midnight UTC that:
  - Calls `this.aggregator.clear()` — NOTE: This is too aggressive (it clears ALL records). Instead, budget reset should NOT clear cost records (Phase 2 cost aggregator is the source of truth). Budget reset means the budget period boundary naturally moves forward. Since `checkBudget()` queries the aggregator with `period: 'day'` which uses `startOfDayUTC()`, when midnight passes, the day period filter automatically excludes yesterday's records. **Budget resets are automatic because the CostAggregator time period filtering already handles this.** The same applies to monthly — `period: 'month'` uses `startOfMonthUTC()`.

  Therefore: **No explicit reset timer is needed.** The CostAggregator's time-windowed queries (`period: 'day'` and `period: 'month'`) already implement daily and monthly resets. Document this in a comment in BudgetEnforcer:
  ```typescript
  // Budget resets are implicit: CostAggregator.getSummary({ period: 'day' })
  // automatically excludes records before today's midnight UTC.
  // No explicit timer or counter reset is needed.
  ```

  However, the `blockedAgents` map (loop cooldowns) needs cleanup. Add a periodic cleanup that removes expired entries. Use `setInterval` with a 60-second interval that removes entries where `expiresAt < Date.now()`. Add `stopCleanup(): void` to clear the interval (for testing).
